Performance Bottleneck Analysis: Why 20 Minutes for a Day?

Problem

Expected: Swisseph can compute a whole day in a fraction of a second
Actual: Taking 20 minutes to compute a whole day

Root Cause Analysis

Current Implementation Flow

For each timestamp in compute_and_store_series():

  1. Call compute_positions() (line 417 in storage.py)
  2. compute_positions() calls compute_subject() (line 1410 in services.py)
  3. compute_subject() creates NEW AstrologicalSubject (line 906 in services.py)
  4. AstrologicalSubject initialization does full computation (Kerykeion overhead)
  5. Extract positions via Subject wrapper and data extraction
  6. Call store_positions() which does individual INSERT

Bottlenecks

1. AstrologicalSubject Creation Overhead (MAJOR)

# This happens for EVERY timestamp!
subj = compute_subject(name, dt_str, loc_str)  # Creates new AstrologicalSubject
# AstrologicalSubject.__init__() does:
#   - Parses datetime
#   - Geocodes location (if needed)
#   - Initializes swisseph
#   - Computes all positions
#   - Builds planets_list, houses, etc.

Problem: Even though swisseph computation is fast, creating the full AstrologicalSubject object has significant overhead:

  • Object initialization
  • Data structure building
  • Multiple method calls
  • Memory allocation

2. Individual INSERT Statements (MODERATE)

# _store_batch() calls store_positions() for EACH timestamp
for datetime_str, positions in positions_batch:
    self.store_positions(...)  # Individual INSERT per timestamp

Problem: Even though batched, each store_positions() call does a separate INSERT statement. Should use executemany() with all rows at once.

3. Subject Wrapper Overhead (MINOR)

# Creates Subject wrapper for each timestamp
subject_wrapper = Subject(name)
subject_wrapper.computed = subj
object_list, degrees_list, labels = subject_wrapper.data()

Problem: Extra object creation and method calls per timestamp.

Performance Math

For 1 day with 1-minute intervals:

  • Timestamps: 1,440 (24 hours Ã— 60 minutes)
  • Current: 20 minutes = 1,200 seconds
  • Per timestamp: ~0.83 seconds

Expected (if swisseph is fast):

  • Swisseph computation: ~0.001 seconds per timestamp
  • Total expected: ~1.4 seconds for 1,440 timestamps

Overhead: ~99.9% of time is overhead, not computation!

Solution: Direct Swisseph Access

Option 1: Use Swisseph Directly (Fastest)

Bypass Kerykeion’s AstrologicalSubject and use swisseph directly:

import swisseph as swe

def compute_positions_swisseph_direct(
    dt: datetime,
    location: Location,
    planets: List[str]
) -> Dict[str, float]:
    """Direct swisseph computation - bypasses Kerykeion overhead."""
    # Convert datetime to Julian Day
    jd = swe.julday(
        dt.year, dt.month, dt.day,
        dt.hour + dt.minute/60.0 + dt.second/3600.0,
        swe.GREG_CAL
    )
    
    # Planet mapping
    planet_map = {
        'sun': swe.SUN,
        'moon': swe.MOON,
        'mercury': swe.MERCURY,
        'venus': swe.VENUS,
        'mars': swe.MARS,
        'jupiter': swe.JUPITER,
        'saturn': swe.SATURN,
        'uranus': swe.URANUS,
        'neptune': swe.NEPTUNE,
        'pluto': swe.PLUTO,
    }
    
    positions = {}
    for planet_name in planets:
        if planet_name in planet_map:
            # Direct swisseph call - very fast!
            xx, ret = swe.calc_ut(jd, planet_map[planet_name], swe.FLG_SWIEPH)
            if ret >= 0:
                longitude = xx[0]  # Ecliptic longitude
                positions[planet_name] = longitude
    
    return positions

Performance: ~0.001 seconds per timestamp (1000x faster!)

Option 3: Batch INSERT Statements (Storage Speedup)

Use executemany() instead of individual INSERTs:

def _store_batch_optimized(self, chart_id: str, positions_batch: List[tuple], 
                          engine: str, ephemeris_file: Optional[str], 
                          radix_chart_id: Optional[str] = None):
    """Store batch using executemany for better performance."""
    rows = []
    for datetime_str, positions in positions_batch:
        dt = datetime.fromisoformat(datetime_str.replace('Z', '+00:00'))
        for object_id, pos_data in positions.items():
            # Build row (same as store_positions but collect all rows first)
            if isinstance(pos_data, dict):
                row = (chart_id, dt, object_id, pos_data['longitude'], ...)
            else:
                row = (chart_id, dt, object_id, float(pos_data), ...)
            rows.append(row)
    
    # Single executemany for all rows
    self.conn.executemany(
        "INSERT OR REPLACE INTO computed_positions (...) VALUES (?, ?, ?, ...)",
        rows
    )

Performance: ~10-100x faster for storage (depends on batch size)

Recommended Solution

Hybrid Approach

  1. For Kerykeion: Use direct swisseph access (Option 1)
  2. For Storage: Use batch INSERT (Option 3)
  3. Keep Kerykeion path: For single chart computations (where overhead is acceptable)

With direct swisseph: ~1.4 seconds for 1 day

Migration Path

  1. Phase 1: Add direct swisseph path alongside existing Kerykeion path
  2. Phase 2: Use direct path for batch computations, Kerykeion for single charts
  3. Phase 3: Optimize storage with batch INSERT
  4. Phase 4: Remove Kerykeion path from batch operations (keep for single charts)
